# Sysmon Configuration Overview

## Purpose & Design Rationale

This configuration maintains separate Sysmon monitoring profiles for **servers**, **endpoints**, and **endpoints with browser logging** to optimize threat detection while managing log volume and system performance:

### Servers Configuration
- **File:** `sysmonconfig-export-servers.xml`
- **Focus:** Network-based attacks, C2 communications, and persistence via scheduled tasks/registry
- **Key Monitors:** DNS queries (Event 22) catch command-and-control beaconing; Registry monitoring detects configuration changes; Limited file monitoring (startup folders only) to reduce noise while catching persistence
- **Trade-off:** Reduced file creation logging in favor of DNS visibility for lateral movement and exfiltration detection

### Endpoints Configuration
- **File:** `sysmonconfig-export-endpoints.xml`
- **Focus:** Malware execution, code injection, and file-based persistence (downloads, temp folders)
- **Key Monitors:** Comprehensive file creation monitoring (Event 11) catches dropped malware and suspicious executables; No DNS monitoring reduces noise on user-facing systems where DNS activity is high
- **Browser tuning:** Firefox, Chrome, Edge, and Brave are excluded from EID 1 (process create), EID 3 (network connect), and EID 5 (process terminate) — browser activity should be monitored via proxy/DNS logs instead
- **Trade-off:** More aggressive file monitoring, less focus on network C2 (handled by network appliances); browser process noise suppressed

### Endpoints with Browser Logging Configuration
- **File:** `sysmonconfig-export-endpoints-browserlogs.xml`
- **Focus:** Identical to the standard Endpoints config with **browser exclusions removed**
- **Use case:** Environments where proxy/DNS logging is not available or not yet deployed, where browser-initiated process creation, network connections, and process exits need to be captured in Sysmon for investigation or threat hunting
- **Key difference from standard Endpoints:** Firefox, Chrome, Edge, and Brave are **not excluded** from EID 1, EID 3, EID 5, EID 8 (CreateRemoteThread target), and EID 12/13 (browser updater registry writes). EdgeWebView AppData file creation is also re-enabled (EID 11)
- **Trade-off:** Significantly higher event volume — browser network connections and child process churn will appear in logs. Expect EID 3 volume increase of 70-90% compared to standard endpoints config. Recommended only as a temporary deployment or for specific high-risk endpoint groups

### Key Differences Explained
| Event | Servers | Endpoints | Endpoints + Browser Logs | Rationale |
|-------|---------|-----------|--------------------------|-----------|
| Event 11 (File Create) | Startup folders only | Full monitoring, Temp/AppData excluded | Full monitoring, Temp/AppData excluded, EdgeWebView re-enabled | Endpoints download/execute more files; servers rely on admin-managed configurations |
| Event 22 (DNS Query) | Monitored | Not monitored | Not monitored | Server C2 is critical detection vector; endpoint DNS is noisy/handled elsewhere |
| Event 4 | Not monitored | Monitored | Monitored | Lower priority for servers; more useful for endpoint troubleshooting |
| EID 1/3/5 Browser | N/A | Excluded (Firefox, Chrome, Edge, Brave) | Included | Browser logs covered by proxy/DNS in standard config; re-enabled when proxy not available |
| EID 3 Network Volume | Low | Medium (browsers excluded) | High (browsers included) | Browser connections are the dominant source of network events on endpoints |

### Registry Load Reduction (Event 12/13/14) - Applied to Both Configs
The following tuning was applied to lower system load from registry monitoring:

**Removed from include rules (high-volume, low detection value):**
- AMCache/InvDB inventory writes (LowerCaseLongPath, Publisher, BinProductVersion, DriverVersion, LinkDate, etc.) — fire on every software execution, minimal attacker coverage
- `HKLM\SYSTEM\CurrentControlSet\Services\` broad wildcard — already fully covered by targeted ServiceDll, ImagePath, and Start rules
- `HKLM\SYSTEM\CurrentControlSet\Enum\Root\LEGACY_` — tagged "unknown log volume" in original config; retained as a re-add candidate after EPS baselining

**Added to exclude rules (trusted high-frequency writers):**
- `TrustedInstaller.exe` — modifies hundreds of keys during Windows patching cycles
- Google/Edge browser updater processes — routine self-update registry churn
- `HKLM\...\FontCache` — routine rendering writes, no attack surface
- Microsoft Monitoring Agent / Splunk agents — known-good management writers
- `HKLM\...\Installer\UserData\` — MSI install/repair cache (broader than existing SYSTEM-only exclude)

**Detection coverage preserved:** All persistence, credential, UAC, Defender tamper, IFEO, COM hijacking, RDP, Winsock, AppCompat shim, HVCI, and trust architecture rules remain intact.

### Event 7 (ImageLoad) Tuning Brief - Applied to Both Configs
Added targeted Sysmon Event ID 7 monitoring to support specific Splunk detections without enabling broad DLL load logging.

**Included (high-signal only):**
- Office processes loading `VBE7INTL.DLL`, `VBE7.DLL`, `VBEUI.DLL`, or `taskschd.dll`
- `taskschd.dll` loaded by binaries executing from suspicious/writable locations (Temp, Public, Fonts, Debug, Recycle Bin, etc.)
- `msi.dll` loads (for non-system module load hunting)
- BRC4-related DLL loads: `credui.dll`, `dbghelp.dll`, `samcli.dll`, `winhttp.dll`

**Excluded (noise reduction):**
- `msi.dll` when the loading process path is under `C:\Windows\*`

**Result:**
- Preserves data required by the referenced Splunk EventCode=7 analytics while keeping endpoint/server load lower than full ImageLoad collection.

### Noise Reduction Exclusions (2026-04-20) — Applied to Both Configs

Data-driven tuning based on analysis of 52,245 sysmon events from endpoint CSV export. Goal: reduce event volume below 10,000/period.

**Trend Micro (all products) — Excluded across EID 1, 3, 5, 7, 10, 12/13, 23, 26:**
Covers ntrtscan.exe, tmbmsrv.exe, TmListen.exe, TmCpmAdapter.exe, isapiclient.dll, HostedAgent, HostedTelemetry, and VAudio (audckq32.exe, wafbsr32.exe). These processes collectively generated ~15,000+ events. Exclusions use broad `begin with` path matching on:
- `C:\Program Files (x86)\Trend Micro\`
- `C:\Program Files (x86)\Common Files\VAudio\`
- Registry keys: `HKLM\SOFTWARE\WOW6432Node\TrendMicro\`, `HKLM\SOFTWARE\Trendmicro\`

Note: EID 4 (Sysmon Service State Changed) is NOT filtered — service shutdown events are still logged.

**Windows Defender (MsMpEng.exe) — Excluded across EID 1, 3, 5, 7, 10, 12/13, 23, 26:**
Routine AV operations generated ~1,000+ registry writes and process events. Exclusions use `begin with` on:
- `C:\ProgramData\Microsoft\Windows Defender\Platform\`
- Registry key: `HKLM\SOFTWARE\Microsoft\Windows Defender\Diagnostics\`

Note: Defender tamper detection rules in EID 12/13 INCLUDE section are preserved (Exclusions key changes, policy changes, disablement detection). The EDR-Freeze detection rule (WerFaultSecure→MsMpEng) in EID 10 INCLUDE is also preserved.

**Snow Software (snowagent.exe) — Excluded across EID 1, 3, 5, 10, 12/13, 23, 26:**
Inventory agent generated ~600+ file delete and process events. Exclusion path: `C:\Program Files\Snow Software\`

**Firefox — Excluded from EID 1, 3, 5:**
Browser generated ~7,400 network events (89% of EID 3), ~850 process creates, ~850 terminations. Network connections should be monitored via proxy/DNS logs instead. Path: `\Mozilla Firefox\firefox.exe`

**Chrome — Excluded from EID 1, 3, 5:**
Same rationale as Firefox. Browser child processes, network connections, and process exits excluded. Path: `\Google\Chrome\Application\chrome.exe`

**Microsoft Edge — Excluded from EID 1, 3, 5:**
Same rationale as Firefox. Path: `\Microsoft\Edge\Application\msedge.exe`

**Brave — Excluded from EID 1, 3, 5:**
Same rationale as Firefox. Path: `\BraveSoftware\Brave-Browser\Application\brave.exe`

**Explorer (Windows shell) — Excluded from EID 1, 5:**
Windows Explorer process creation and termination events are routine shell noise. Path: `C:\Windows\explorer.exe`

**Cisco Secure Client — Excluded from EID 26:**
VPN/Umbrella agent generated ~305 file delete events. Path: `C:\Program Files (x86)\Cisco\Cisco Secure Client\`

**IFEO MitigationOptions/MitigationAuditOptions — Excluded from EID 12/13:**
taskhostw.exe constantly rewrites Windows exploit protection mitigation settings under `Image File Execution Options\*`. ~9,200 events (80% of EID 13). Excluded by `end with` on `\MitigationOptions` and `\MitigationAuditOptions`.

**svchost.exe → lsass.exe — Excluded from EID 10:**
Normal Windows authentication/token operations. 3,714 events (63% of EID 10).

**conhost.exe — Excluded from EID 5:**
Console host terminations are pure noise. 897 events. (Already excluded from EID 1 in base config.)

**Estimated impact:** ~52,245 → ~8,000-10,000 events (80-85% reduction)

**Detection coverage preserved:**
- All INCLUDE rules unchanged (persistence, credential theft, LSASS access with suspicious permissions, COM hijack, IFEO debugger, Defender tampering, ransomware file deletion, CobaltStrike BOF patterns, EDR-Freeze, etc.)
- EID 4 (service state) unfiltered — still logs Sysmon/Defender service stops
- Defender registry tamper rules in EID 12/13 include section untouched

### EVENT 11 (File Created) - Temp/AppData Exclusions Tuning (2026-04-29) — Change #3 - Applied to Both Configs

Data-driven tuning based on analysis of 18,285 sysmon events from 24-hour endpoint logs. Focus: reducing noise from known-good application caching/temp operations while preserving persistence and malware detection.

**Problem Identified:**
- EVENT 11 represents 10.1% of all events (1,851 events)
- 54% of file creations are in Temp/AppData/Cache locations for benign applications
- Top noisy sources:
  - Microsoft Office Hub (WindowsApps): 330 events (17.8% of EVENT 11)
  - Trend Micro Security Agent temp folder: 143+ events (7.7% of EVENT 11)
  - Microsoft AETP DataCollection: 57 events (3.1% of EVENT 11)

**Solution Implemented - Endpoints Config:**
Added 10 targeted exclusion rules to the EVENT 11 `exclude` RuleGroup:
1. `\AppData\Local\Temp\` — General user temporary folder (non-executable context, monitored by antivirus)
2. `\AppData\Local\Microsoft\Windows\Temporary Internet Files\` — IE cache folder
3. `\AppData\Local\Microsoft\EdgeWebView\` — Edge WebView runtime updates (193 baseline events)
4. `\AppData\Local\Programs\Microsoft Visual Studio Code\` — VS Code self-updates and cache
5. `\AppData\Local\Microsoft\OneDrive\` — OneDrive sync cache (already monitored by Microsoft)
6. `\AppData\LocalLow\` — Low-integrity storage (non-executable)
7. `C:\Program Files (x86)\Trend Micro\Client Server Security Agent\Temp\` — TMCSSA maintenance (143+ baseline events)
8. `C:\ProgramData\Microsoft\Windows Defender Advanced Threat Protection\DataCollection\` — AETP telemetry (57 baseline events)
9. `C:\Program Files\WindowsApps\` — UWP/AppX package updates and caching (330+ baseline events)

**Solution Implemented - Servers Config:**
Added 4 targeted exclusion rules to the EVENT 11 `exclude` RuleGroup (servers have minimal FILE CREATION monitoring):
1. `\AppData\Local\Temp\` — General temporary folder
2. `\AppData\Local\Microsoft\Windows\Temporary Internet Files\` — IE cache
3. `C:\ProgramData\Microsoft\Windows Defender Advanced Threat Protection\DataCollection\` — AETP telemetry
4. `C:\Program Files\WindowsApps\` — UWP/AppX updates

Note: Servers retain their narrow focus on Startup folder monitoring (persistence detection) and Startup Menu shortcuts. This tuning prevents noise without changing detection strategy.

**Impact Estimate:**
- Endpoints: 1,851 EVENT 11 events → ~1,000-1,100 remaining (~45% reduction)
- Servers: Minimal impact (servers already filter to startup folders only)
- Overall log volume: 18,285 → ~17,500 events (4-5% reduction)

**Detection Capability Preserved:**
- All INCLUDE rules for EVENT 11 remain unchanged: suspicious file extensions (.exe, .ps1, .dll, .bat, .scr, .hta, etc.), Downloads folder monitoring, Startup persistence detection, admin-initiated UAC bypass tools (.aspx, .ashx webshells), LSASS credential dumps, SAM registry extractions, ClickOnce artifacts, system driver installation folders
- Temporary exclusions are LIMITED TO known-good, vendor-signed applications only
- Malware dropped to Temp folder by suspicious processes will still be logged via parent process analysis (EID 1) and subsequent execution (EID 1, 11 parent chain)
- No exclusion applies to user DOWNLOADS folder or Startup folders — critical persistence vectors remain monitored

**Rationale for Each Exclusion:**
- Temp/Cache folders: Non-executable context; temporary; removed on cleanup
- VS Code/Edge/OneDrive: Vendor-signed, regularly updated, benign data types
- Trend Micro Agent temp: Maintenance operations, non-security-relevant writes, already monitored by Trend Micro themselves
- WindowsApps folder: UWP package store, Microsoft-managed, signed binaries only
- AETP DataCollection: System telemetry, non-security-relevant, high volume

**Future Tuning Opportunity:**
- Monitor Trend Micro HostedAgent process-creation patterns and File Deletes separately if suspicious file-drop behavior is identified. These exclusions can be refined by exception rules if needed.
- Consider environment-specific additions: Slack, Teams, or other collaboration apps' AppData folders if similar baseline analysis identifies them as high-volume, low-signal sources.

TOOLING-SPECIFIC TUNING - SCCM, ManageEngine, Nessus, Veeam (2026-04-29)
================================================================================

**Summary:**
Pre-emptive exclusions for four common enterprise tools not present in the 24-hour baseline but likely deployed in customer environments. Applied to all three config files. No baseline event count available — volume estimates based on known agent behaviour patterns.

| Tool | Event IDs Tuned | Notes |
|------|----------------|-------|
| SCCM/ConfigMgr | EID 1, 5, 7, 10, 11, 12/13 | Highest priority — extremely common, very noisy during patch windows |
| ManageEngine Endpoint Central | EID 1, 5, 7, 12/13 | Formerly Desktop Central; two install path variants covered |
| Tenable Nessus Agent | EID 1, 5 | Only process create/terminate excluded; EID 3 retained (scanner traffic has detection value) |
| Veeam | EID 1, 5, 11 | EID 11 scoped to ProgramData path only — backup agent writing to unusual paths should still fire |

---

**SCCM/ConfigMgr (Microsoft Configuration Manager)**
Install paths: `C:\Windows\CCM\`, `C:\Windows\ccmsetup\`

- **EID 1/5**: Image + ParentImage for CCM\ and ccmsetup\ — SCCM spawns many short-lived child processes (scripts, installers, inventory collectors) every scan cycle
- **EID 7**: `C:\Windows\CCM\` image DLL loads — CCM client loads many DLLs during policy evaluation
- **EID 10**: `C:\Windows\CCM\` as SourceImage — CCM performs process enumeration for software inventory
- **EID 11**: TargetFilename paths `C:\Windows\ccmcache\`, `C:\Windows\ccmsetup\`, `C:\Windows\CCM\Logs\` — software deployment staging and log rotation generate sustained file-create noise
- **EID 12/13**: CCM image + `HKLM\SOFTWARE\Microsoft\CCM\` and `HKLM\SOFTWARE\Microsoft\SMS\` key paths — client policy and state writes

Note: SCCM is likely the largest single source of untuned noise in any ConfigMgr-managed environment. During a patch Tuesday deployment window, SCCM can generate 2,000–5,000 additional events per endpoint without these exclusions.

---

**ManageEngine Endpoint Central (formerly Desktop Central)**
Install paths: `C:\Program Files (x86)\DesktopCentral_Agent\`, `C:\Program Files (x86)\ManageEngine\UEMS_Agent\`

Both path variants are covered — older deployments use DesktopCentral_Agent; newer UEMS-based deployments use ManageEngine\UEMS_Agent. Both Image and ParentImage included for EID 1; Image-only for EID 5, 7, 12/13.

- **EID 7**: Agent loads its own DLL stack during patch scanning cycles
- **EID 12/13**: Agent writes to its own registry hive during policy sync

---

**Tenable Nessus Agent**
Install path: `C:\Program Files\Tenable\Nessus Agent\`

- **EID 1/5 only** — process create/terminate excluded (scanner spawns many short-lived probe processes during scan jobs)
- **EID 3 intentionally NOT excluded** — network connections from a vulnerability scanner to other hosts are high-signal events. Lateral scan traffic should be visible even if Nessus is authorized.
- **EID 7/10 not excluded** — Nessus reads process memory as part of local checks; this is security-relevant and should remain visible.

---

**Veeam (Backup & Replication)**
Install path: `C:\Program Files\Veeam\`

- **EID 1/5**: Agent and job runner process churn during backup jobs (Image + ParentImage)
- **EID 11 scoped**: Only `C:\ProgramData\Veeam\` TargetFilename excluded — this covers Veeam's own working directory (job metadata, temp staging, VSS coordination files). The EID 11 Image-based exclusion is intentionally NOT used so that Veeam writing files outside its own ProgramData directory (e.g., to user shares or system paths during a ransomware-impersonated backup process) would still be detected.

**Detection Capability Preserved:**
- Nessus network scanning traffic remains visible in EID 3
- Veeam writing executables or scripts to non-ProgramData paths remains visible in EID 11
- SCCM software deployment to monitored paths (Downloads, Startup, etc.) remains visible via EID 11 include rules
- None of these exclusions apply to EID 4 (service state), EID 8 (remote thread), EID 22 (DNS), or pipe events

**CHANGE #8 — MDE MsSense.exe EID 11 (File Create)**
Target: `MsSense.exe` (Microsoft Defender for Endpoint sensor)
Event IDs: 11 (FileCreate)

Added `<Image condition="image">MsSense.exe</Image>` to the EID 11 exclude block alongside the existing AETP DataCollection path exclusion.

Rationale: MsSense.exe writes its own sensor telemetry to the DataCollection path. The path-based rule already covered these files, but the Image rule adds a broader catch in case the sensor writes to adjacent paths on newer MDE versions. Low security value — MDE sensor file activity is self-monitored by MDE itself.

---

**CHANGE #6 — Google Updater EID 1, 5, 7 (Process Create/Terminate, Image Load)**
Target: `C:\Program Files\Google\GoogleUpdater\` and `C:\Program Files (x86)\Google\GoogleUpdater\`
Event IDs: 1, 5, 7

Note: Existing config already excluded `C:\Program Files\Google\Update\` (old updater path) from EID 12/13. The new GoogleUpdater binary uses a different path — these are separate executables. EID 1/5/7 had no exclusion for either path.

Rules added:
- EID 1: Image + ParentImage begin with GoogleUpdater\ paths
- EID 5: Image begin with GoogleUpdater\ paths
- EID 7: Image begin with GoogleUpdater\ paths (ImageLoad exclude RuleGroup)

Rationale: Google's background update service fires hourly with multiple short-lived child processes and DLL loads. No lateral movement or injection risk from a signed Google binary in Program Files. Existing EID 12/13 exclusion for the old path already established precedent.

---

**CHANGE #3 — MDM Agents EID 1, 12/13 (Process Create, Registry Event)**
Target: `deviceenroller.exe` and `omadmclient.exe` (Windows built-in Intune/MDM enrollment agents)
Event IDs: 1, 12/13

Rules added:
- EID 1: `<Image condition="image">deviceenroller.exe</Image>` and `omadmclient.exe`
- EID 12/13: Same Image rules, plus a scoped WMIADAP rule (see below)

Note: EID 1 ProcessCreate include already has a CommandLine include for `deviceenroller.exe /c /AutoEnrollMDM` as a high-signal detection rule. The new EID 1 exclude only suppresses non-enrollment executions of these binaries (the include takes priority over the exclude for that specific CommandLine pattern).

Rationale: These are Windows built-in MDM agents that fire periodically on any Intune-managed or domain-joined device. Registry writes include `HKLM\SOFTWARE\Microsoft\Enrollments\` and `HKLM\SOFTWARE\Microsoft\PolicyManager\` — high volume, zero attack surface from SYSTEM-owned built-in binaries.

Also added in this change (grouped for implementation):

**WMIADAP scoped EID 12/13:**
```xml
<Rule groupRelation="and">
    <Image condition="image">WMIADAP.EXE</Image>
    <TargetObject condition="begin with">HKLM\SOFTWARE\Microsoft\Wbem\</TargetObject>
</Rule>
```
Rationale: WMIADAP (WMI Performance Counter Adapter) runs every ~15 minutes and writes exclusively to the Wbem performance counter key. The scoped AND rule prevents blanket exclusion of the process — only its known-good write path is suppressed. WMI persistence writes to other paths (e.g., HKLM\SOFTWARE\Microsoft\WBEM\ESS) will still be logged.

---

**CHANGE #2 — Microsoft EdgeWebView2 EID 1, 5, 10 (Process Create/Terminate, Process Access)**
Target: `msedgewebview2.exe` (Microsoft Edge WebView2 embedded browser runtime)
Event IDs: 1, 5, 10

Note: EdgeWebView2 is NOT a standalone browser — it is the embedded rendering runtime used by Outlook, Teams, Visual Studio Code, and other Microsoft applications. It should be excluded even in the `endpoints-browserlogs` variant (which strips standalone browser exclusions). This is why it is included in all three configs.

Rules added:
- EID 1: `<Image condition="end with">\msedgewebview2.exe</Image>` and `<ParentImage condition="end with">\msedgewebview2.exe</ParentImage>`
- EID 5: `<Image condition="end with">\msedgewebview2.exe</Image>`
- EID 10: Scoped AND rule: Outlook.exe → msedgewebview2.exe (normal embedded rendering)
- EID 10: Scoped AND rule: IntuneWindowsAgent.exe → agentexecutor.exe (normal management spawning)

Rationale: EdgeWebView2 spawns dozens of short-lived child processes during every Outlook/Teams session. EID 1 and EID 5 for this binary represent ~366 events per 24 hours. EID 10 access from Outlook to EdgeWebView2 is the normal rendering model for Outlook's embedded HTML content. No security value in monitoring a SYSTEM/USER signed Microsoft binary that is child of a signed Microsoft host process.
